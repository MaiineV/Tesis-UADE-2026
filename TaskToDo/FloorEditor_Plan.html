Floor Editor — Plan & Visual Mockup

Herramienta de Editor para autorear FloorLayoutSOs sin min/max y con pool por tipo de sala con búsqueda.

1. Objetivo

Reemplazar la edición actual de FloorLayoutSO (vía Inspector + Odin) por una ventana dedicada Floor Editor que permita:

  • Definir cuántas salas tiene un piso y de qué tipo exacto.
  • Por cada RoomType (Combat, Shop, Potion, Boss, Start), un contador que puede ser Fixed (valor exacto, ej. 3) o Random (rango Min..Max, ej. 2..5 — se rerollea al generar el piso).
  • Por cada RoomType, una pool de candidates (lista de RoomSO) configurable con dropdown + búsqueda sobre todas las RoomSO del proyecto, filtrada por tipo.
  • Vista previa de topología sintética (mini-grilla) que muestre cómo quedaría el piso resultante.

2. Cambios de Data (FloorLayoutSO)

El SO se reescribe completamente. Se eliminan todos los campos viejos (RoomCountMin/Max, CombatRooms, ShopRooms, PotionRooms, DefaultBossRoomTemplate, BossCandidates, StartRoom) y el asset existente se rearma a mano una sola vez. Sin migrator, sin código de compatibilidad, sin atributos [Obsolete]. 

// Modelo final — único formato soportado
public enum RoomCountMode { Fixed, Random }

[Serializable]
public class RoomCountSpec {
    public RoomCountMode Mode;      // Fixed o Random
    public int Fixed;               // usado si Mode == Fixed
    public int Min;                 // usado si Mode == Random
    public int Max;                 // usado si Mode == Random

    public int Resolve(System.Random rng) =>
        Mode == RoomCountMode.Fixed ? Fixed : rng.Next(Min, Max + 1);
}

[Serializable]
public class RoomTypeSlot {
    public RoomType Type;           // Combat / Shop / Potion / Boss / Start
    public RoomCountSpec Count;     // Fixed o Random
    public List<RoomSO> Pool;      // pool de candidates para este tipo
}

public class FloorLayoutSO : SerializedScriptableObject {
    public string FloorId;
    public string DisplayName;
    public List<RoomTypeSlot> Slots; // 1 entry por RoomType usado
}

Notas de UX:
  • Cada Slot es una tarjeta colapsable color-coded por tipo.
  • El header del slot tiene un toggle Fixed/Random. Fixed muestra un stepper de 1 valor; Random muestra dos inputs Min..Max (el editor fuerza Min ≤ Max).
  • Tanto Fixed como Min/Max admiten 0 — el slot queda "tipo posiblemente ausente" pero conserva su pool guardada.
  • El botón + Add Room… abre un search dropdown que lista todas las RoomSO del proyecto, con el tipo correcto resaltado y los demás disabled (filter visual, no hard block — el usuario puede insistir).
  • El preview de la derecha corre el algoritmo de topología contra un seed configurable, resolviendo cada RoomCountSpec con ese seed. Sólo visualización — no se persiste.
  • El status bar valida en vivo: pool vacío + count>0 es error; Min=Max=0 con pool poblada es warn; Min>Max es error.

4.3 RoomSearchDropdown — algoritmo

  1. Al abrir: cargar todos los RoomSO con AssetDatabase.FindAssets("t:RoomSO") (cacheado por sesión, invalidado en OnProjectChange).
  2. Render con EditorGUILayout.TextField para búsqueda en tiempo real (matching case-insensitive sobre RoomId + DisplayName).
  3. Filtrado visual por RoomType: matching = enabled & ordenado primero; non-matching = disabled & atenuado al final.
  4. Click en row → Undo.RecordObject(floor, "Add Room to Pool") + slot.Pool.Add(room) + EditorUtility.SetDirty + close dropdown.
  5. No permite duplicados (el room ya presente en la pool aparece tachado o ausente).

4.4 Cambios en DungeonManager (reescritura completa de AssignTemplates)

Se borran los campos legacy y los paths que los leen. El método nuevo:

  1. Resuelve cada slot.Count con RoomCountSpec.Resolve(rng) (Fixed devuelve Fixed, Random devuelve rng.Next(Min, Max+1)).
  2. Calcula targetCount = Σ resolvedCounts. Esto reemplaza al viejo RoomCountMin/Max — la topología random walk genera exactamente esa cantidad de cells.
  3. Asigna templates por cell con esta prioridad:
    • Start → Vector2Int.zero, template random de la pool.
    • Boss → las N cells más lejanas en Manhattan distance.
    • Shop / Potion / otros tipos especiales → cells intermedias random hasta llenar su resolvedCount.
    • Combat → cells restantes.
  4. Si la topología no logra targetCount cells (frontera bloqueada), se loggea warning y los slots de menor prioridad se truncan (Combat primero).

6. Preguntas Abiertas — Confirmar antes de codear

  • ¿Múltiples Boss por piso permitidos? Hoy hay 1 fijo. El stepper/range permite N — si se mantiene "1 boss = final del piso", el evento OnFloorCleared debe esperar a que todas las bosses estén cleared. Sugiero permitirlo y que el evento dispare al matar la última.
  • ¿Slot con count=0 / Min=Max=0 se conserva o se elimina al guardar? Sugiero conservar — útil para alternar tipos sin perder la pool autoreada.
  • ¿La pool admite duplicados? Hoy el mismo RoomSO puede aparecer 2× en CombatRooms. Recomiendo bloquear duplicados en el dropdown — para "más probabilidad" usar un WeightedEntry en el futuro.
  • ¿Preview topológico es read-only o editable? Read-only es más simple — usa el seed configurable para mostrar un roll concreto del RoomCountSpec. Editable (drag rooms) implica capa de manual placement — out of scope.
  • ¿El seed del preview es persistente (campo del SO) o sólo de la ventana? Sugiero sólo de la ventana — el seed real en runtime lo elige el Run.
